# Codebase Review: Stellation
**Date:** 2026-07-15
**Stack:** FastAPI (Python 3.12) + pyswisseph/numpy/matplotlib/geopy/timezonefinder/anthropic — Vite + React 19 + TypeScript
**Project type:** Standalone full-stack web app (generative natal chart art + AI interpretation)

## Executive Summary
This is a tightly-scoped, well-crafted codebase — the domain math (ephemeris, houses, aspects) is isolated into pure service functions, comments consistently explain *why* rather than *what*, and the frontend/backend split cleanly matches the "new learning vs. known execution" plan from the spec doc. The biggest gaps aren't code quality issues, they're production-readiness gaps: the two Anthropic-calling endpoints have no abuse protection despite being live on a public Vercel URL, and a couple of error-handling edge cases can fail silently instead of showing the user anything. Neither is hard to fix.

## High Priority Findings

### 1. `/api/interpret` and `/api/synastry/interpret` have no rate limiting or abuse protection
**What:** Both endpoints (`backend/app/routers/interpret.py`, `backend/app/routers/synastry.py`) call the Anthropic API on every request with no auth, no per-IP throttling, and no request cap. CORS restricts which *browsers* can call them from JS, but CORS does nothing against a direct script/curl hitting the deployed URL.
**Why it matters:** This app is deployed to a real Vercel URL with a billed `ANTHROPIC_API_KEY`. Anyone who finds the URL can script unlimited calls and run up your Anthropic bill — there's currently nothing between "someone finds the link" and "unbounded spend."
**Suggested action:** Add lightweight per-IP rate limiting (e.g. `slowapi` or a simple in-memory/token-bucket dependency) on the two `/interpret` routes specifically — those are the only ones with a real dollar cost per call. `/chart` and `/render` are free to leave open.
**Effort estimate:** Half-day

### 2. Malformed/unexpected input can fail silently on the frontend
**What:** Two compounding gaps:
- `ChartRequest.birth_date` / `birth_time` (`backend/app/models/schemas.py`) are plain `str` fields with no format validation — a value that doesn't match `YYYY-MM-DD` / `HH:MM` reaches `dt.datetime.strptime` in `ephemeris.py:local_to_jd_ut` unguarded and raises an uncaught `ValueError`, producing FastAPI's generic 500 (`{"detail": "Internal Server Error"}`).
- `frontend/src/api.ts`'s error parsing (`fetchChart`, `fetchSynastry`) assumes `body.detail` is always the app's own `{error, message}` shape. When the backend instead returns a plain string (generic 500) or an array (Pydantic 422 validation errors), `detail` becomes that string/array, and `App.tsx`'s `setErrorMessage(err.detail.message)` reads `.message` off a string — which is `undefined`. Since `{errorMessage && ...}` is falsy on `undefined`, **no error renders at all**: the submit button just goes back to idle with no feedback.
**Why it matters:** Right now this can't be triggered through the normal form (native `<input type="date">`/`type="time">` always produce valid formats), but it's one `curl` away, and it means the error path has a real blind spot rather than degrading gracefully.
**Suggested action:** Add a `field_validator` (or regex `pattern`) on `birth_date`/`birth_time` in `schemas.py` so bad input becomes a clean 422 with the app's own error shape instead of a crash. Separately, harden `api.ts`'s error parsing to fall back to the generic message whenever `body.detail` isn't an object with a `message` field.
**Effort estimate:** Quick fix (1-2 hours)

## Medium Priority Findings

### 3. TypeScript `strict` mode isn't enabled
**What:** Neither `tsconfig.app.json` nor `tsconfig.node.json` sets `"strict": true`. Only `noUnusedLocals`/`noUnusedParameters`/`noFallthroughCasesInSwitch` are on — no strict null checks, no `noImplicitAny`.
**Why it matters:** Your own convention is "TypeScript always... Error handling: always handle, never swallow silently" — strict mode is what actually enforces that at the type level (e.g. it would have caught the `err.detail.message` issue above at compile time, since `detail` would need a narrower type). The codebase is clean enough right now that turning it on is likely low-friction.
**Suggested action:** Add `"strict": true` to `tsconfig.app.json`, run `tsc -b`, fix whatever surfaces.
**Effort estimate:** Quick fix (given how clean the current code is)

### 4. No test coverage on the calculation core
**What:** No test files anywhere in the repo. The app's entire value proposition per the README is "a real natal chart... nothing hand-waved" — the pure-function math lives in `aspects.py` (orb/applying-separating logic) and `ephemeris.py` (`_sign_and_degree`, `_house_of`), both easy to unit test since they take plain floats/dicts and have no I/O.
**Why it matters:** A sign-off-by-one or a house-boundary regression would silently ship wrong readings with nothing to catch it — the kind of bug that's invisible until a user notices their Sun sign is wrong.
**Suggested action:** A focused `pytest` suite on `_sign_and_degree`, `_house_of`, and `compute_aspects`/`compute_synastry_aspects` (a handful of known-answer cases: boundary degrees, retrograde speed sign, orb edge cases) would cover the highest-risk surface without a big testing investment.
**Effort estimate:** Half-day

### 5. No way to save/download the generated chart art
**What:** The chart art renders as an `<img>` pointing at a blob URL (`ChartCarousel.tsx`) with no explicit save/download control.
**Why it matters:** The generative art is the emotional payoff of the whole flow — someone fills out a form, waits through the `GeneratingScreen`, and gets a unique piece of art they can't cleanly keep. Right-click-save works but is undiscoverable, especially on mobile where it's not even available the same way.
**Suggested action:** Add a "Download" link/button next to the carousel using the existing blob URL (`<a href={slide.url} download={...}>`).
**Effort estimate:** Quick fix

### 6. `missing_location` error doesn't reveal the manual-coordinates fields
**What:** In `App.tsx`, `handleSoloSubmit`/`handleSynastrySubmit` only call `setShowManualCoords(true)` when `err.detail.error === 'geocode_failed'`. If a user submits with an empty birth place (the field isn't marked `required`), the backend raises `missing_location` instead (`chart_builder.py`) — same "enter coordinates manually" message, but the manual lat/lng fields never appear.
**Why it matters:** The user isn't fully stuck (they can just type a place name instead), but the error message tells them to do something the UI doesn't let them do.
**Suggested action:** Either mark `birth_place` as required client-side (simplest), or also flip `showManualCoords` on `missing_location`.
**Effort estimate:** Quick fix

## Low Priority / Nice-to-Have

### 7. `useChartReveal` and `useSynastryReveal` are near-duplicates
Same state shape, same reset-during-render pattern, same settled/generating logic — only the fetch calls differ. Could be one generic hook parameterized by fetch functions. Pure simplification, not a bug.
**Effort estimate:** Quick fix

### 8. Carousel controls are small touch targets
`.carousel-arrow` is 32px (`App.css:381-382`) and `.carousel-dot` is 7px — both below the ~44px comfortable tap-target guideline, and they're the *only* way to compare generative vs. traditional art style on mobile.
**Effort estimate:** Quick fix (pad the hit area, keep the visual size)

### 9. No backend linter/formatter
Frontend has `oxlint` configured; backend has no `ruff`/`black`/equivalent. Minor DX inconsistency between the two halves.
**Effort estimate:** Quick fix

### 10. Geocoding isn't cached
Every chart submission re-hits Nominatim even for a repeated place name. Fine at current traffic, and Nominatim's usage policy (≤1 req/sec) isn't a concern at this scale, but a simple in-memory cache would be a one-line win if usage ever grows.
**Effort estimate:** Quick fix

## Dimension Summaries

### UI/UX: 🟡 Clean and accessible, with a few rough edges
Focus-visible outlines, `prefers-reduced-motion` support, `aria-live="polite"` on the generating screen, shared form components between solo/synastry — all solid, ND-conscious defaults. Gaps are the small touch targets (#8), the missing-location dead-end (#6), and no art export (#5).

### Features: 🟡 Core loop is complete; a few QoL gaps
Solo chart, synastry, two art styles, zodiac/house-system options, relationship-aware synastry framing — all fully wired end-to-end, no orphaned backend/frontend halves found. Biggest gap is no save/export path for the art (#5); a secondary one is that birth time is always required with no "I don't know my exact birth time" fallback, which is a real barrier for some users (worth a product decision, not flagging as a defect).

### MCP: N/A by design
The README/spec explicitly scope this out — standalone project, no shared ecosystem integration. Nothing to flag.

### Architecture: 🟢 Clean separation, one real gap
Routers/services/models split cleanly, pure functions for the domain math, no obvious layering violations. The `strict` mode gap (#3) is the one thing worth closing given how much this project leans on TypeScript's guarantees per your conventions.

### Infrastructure: 🟡 Deployment story is solid; cost exposure isn't handled
The Vercel Services setup (shared domain, same-origin `/api/*`, single `ANTHROPIC_API_KEY` env var) is a clean way to avoid CORS complexity in prod. The real gap is #1 — no protection against a public URL turning into unbounded API spend.

### DX: 🟡 Good in-code documentation, no safety net
Comments consistently capture non-obvious *why* (Moshier ephemeris choice, forced tool-use for model portability, sidereal/tropical house-cusp alignment) — genuinely good documentation-in-code. No automated tests (#4) and no CI mean none of that correctness is actually verified by anything other than manual testing.

## Strengths
- **Comments explain the why, never the what** — the Moshier ephemeris rationale, the forced-`tool_choice` model-portability note, the sidereal house-cusp alignment note, the synastry ripple-ratio geometry note. This is exactly the kind of comment worth keeping as the codebase grows.
- **Domain math is cleanly isolated** — `aspects.py` and `ephemeris.py` are pure functions over plain data, which makes them trivial to test (see #4) and easy to reason about independently of FastAPI/Pydantic.
- **Real component reuse** — `PersonFields` and `ChartSettingsFields` are shared verbatim between the solo and synastry forms rather than duplicated, and the synastry render/interpretation code reuses the natal chart's orbit-ring rendering logic instead of building a second visual language from scratch.
- **Palette discipline** — the exact same hex values drive both the UI (`index.css`) and the generated chart art (`render.py`), explicitly called out in a comment, so the app and its output read as one object.
- **Error UX for the geocoding-failure path** (the *working* case) is genuinely thoughtful — the message plus the conditional manual-coordinate fallback fields is a good pattern, just needs the `missing_location` branch to match it (#6).
